Une question de cycles

The cycles of TDD by Uncle Bob

Nano cycles

  1. You must write a failing test before you write any production code.
  2. You must not write more of a test than is sufficient to fail, or fail to compile.
  3. You must not write more production code than is sufficient to make the currently failing test pass.

Micro cycles

  1. Create a unit test until have one that fails
  2. Write production code that makes that test pass.
  3. Clean up the mess you just made.

Étape 1 - Ajouter un test

Partir d'une page blanche permet de mieux s'abstraire des contraintes pour comprendre le comportement attendu.

Pour passer à l'étape suivante, ce test doit :

  • Échouer
  • Être le seul test en échec

Étape 2 - Écrire le code

Faire passer le test, peu importe la manière.

  • Fake it: retourner une valeur constante ...
  • Triangulation: 2 exemples avant généralisation
  • Implémentation évidente: implementation réelle pour les cas simples
  • One to many: pour des listes, implémenter pour un cas puis pour plusieurs

Exécuter tous les tests, jusqu'à ce que le dernier test ajouté passe.

Étape 3 - Refactoring

Améliorer le code produit sans en changer le comportement.

  • DRY, YAGNI, SRP, KISS, SOLID ... Clean Code !
  • Généralisation du code ou des comportements similaires
  • Abstractions
  • Applicable sur le code de production et sur les tests
